A Ticto API v2 tem dois ambientes **independentes**: **sandbox** (testes) e **produção** (real). Eles têm o mesmo contrato: os mesmos endpoints, os mesmos campos, as mesmas respostas. A única diferença prática são a **base URL** e o **prefixo do token**. Você desenvolve inteiro no sandbox e, quando estiver pronto, troca duas coisas para ir ao ar.

## Sandbox vs. Produção

|  | Sandbox | Produção |
|---|---|---|
| **Base URL** | `https://sandbox.ticto.cloud` | `https://api.ticto.cloud` |
| **Prefixo do token** | `ticto_test_` | `ticto_live_` |
| **Cobranças reais** | Não (dinheiro fictício) | Sim |
| **Produtos visíveis ao público** | Não | Sim, por padrão (`is_active: false` adia a publicação) |
| **Dados** | Isolados e descartáveis | Permanentes |

<Note>
  Os ambientes são separados de ponta a ponta. Um token `ticto_test_` só funciona em `sandbox.ticto.cloud` e um `ticto_live_` só funciona em `api.ticto.cloud`. Não há como um token de teste tocar dados reais.
</Note>

## Como obter acesso

O fluxo de gerar o token é idêntico nos dois ambientes. Muda apenas onde você entra:

<Steps>
  <Step title="Sandbox para desenvolver">
    Acesse o **painel sandbox** da Ticto, vá em **Integrações → API v2** e gere um token. Ele virá com o prefixo `ticto_test_`. Use-o contra `https://sandbox.ticto.cloud`.
  </Step>
  <Step title="Produção para ir ao ar">
    No **painel de produção**, no mesmo lugar (**Integrações → API v2**), gere um token `ticto_live_`. Use-o contra `https://api.ticto.cloud`.
  </Step>
</Steps>

O prefixo é sua bússola: olhando o token você sabe na hora se está falando com teste ou com produção: em código, em logs e em variáveis de ambiente. O token completo é exibido **uma única vez** na criação; guarde-o com segurança.

## Migrando do sandbox para a produção

Como o contrato é o mesmo, ir ao ar é trocar **duas variáveis**:

```diff
- BASE_URL=https://sandbox.ticto.cloud
- TICTO_TOKEN=ticto_test_xxxxxxxx
+ BASE_URL=https://api.ticto.cloud
+ TICTO_TOKEN=ticto_live_xxxxxxxx
```

Nenhuma mudança de payload, de rota ou de lógica. O que passou no sandbox passa em produção.

## Disponibilidade por padrão

Em **qualquer** ambiente, todo produto criado via API nasce com status `pre-approved` (pré-aprovado) e **ativo** (`is_active: true`), pronto para vender: após a primeira venda ele passa pela análise de compliance. O `status` é da plataforma e não pode ser definido pela API. Se você quiser preparar o produto antes de divulgá-lo, crie-o com `is_active: false` e ative-o depois no painel.

<Tip>
  Recrie o mesmo fluxo de ponta a ponta no sandbox antes de virar a chave. Assim você valida token, escopos, criação e leitura sem risco, e a virada para produção vira só a troca das duas variáveis acima.
</Tip>
