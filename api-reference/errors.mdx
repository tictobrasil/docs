Todos os erros da Ticto API v2 retornam um corpo JSON padronizado com slugs estáveis no campo `error`. Isso permite que sua integração trate erros de forma programática e confiável, independentemente de mudanças nas mensagens descritivas.

## Envelope de erro

Toda resposta de erro segue a estrutura abaixo:

```json Envelope de erro
{
  "error": "validation_error",
  "message": "Os dados enviados contêm inconsistências. Verifique o campo details.",
  "request_id": "req_7f3a91bc4d2e",
  "details": [
    {
      "field": "price",
      "message": "O preço deve ser um número inteiro em centavos e no mínimo 500."
    },
    {
      "field": "recurrence.interval",
      "message": "O intervalo informado não é válido. Use: monthly, quarterly, semiannually ou annually."
    }
  ]
}
```

| Campo        | Tipo            | Descrição                                                                                         |
|--------------|-----------------|---------------------------------------------------------------------------------------------------|
| `error`      | `string`        | Slug estável do erro. Use este campo para lógica de tratamento no seu código.                     |
| `message`    | `string`        | Descrição legível do erro. Pode ser exibida ao usuário ou usada para debugging.                   |
| `request_id` | `string`        | Identificador único da requisição. Inclua este valor ao abrir tickets de suporte na Ticto.        |
| `details`    | `array`         | Lista de erros de validação por campo. Presente somente em erros `422`. Pode ser um array vazio.  |

## Tabela de erros

| Status HTTP | Slug do erro           | Quando ocorre                                                                                   |
|-------------|------------------------|-------------------------------------------------------------------------------------------------|
| `400`       | `bad_request`          | Corpo da requisição mal formatado, JSON inválido ou parâmetros com tipo incorreto.              |
| `401`       | `unauthenticated`      | Token de autenticação ausente, malformado ou expirado.                                          |
| `403`       | `forbidden`            | Token válido, mas sem permissão (escopo) para acessar o recurso solicitado.                     |
| `404`       | `not_found`            | O recurso solicitado (produto, oferta, etc.) não existe ou não pertence à sua conta.            |
| `409`       | `conflict`             | A requisição conflita com o estado atual — geralmente uma `Idempotency-Key` já utilizada.       |
| `422`       | `validation_error`     | Os dados enviados falharam na validação. Consulte o array `details` para os campos com erro.    |
| `429`       | `too_many_requests`    | Limite de requisições por token atingido. Use o header `Retry-After` para aguardar o reset.     |
| `500`       | `internal_server_error`| Erro inesperado no servidor. Se persistir, abra um ticket com o `request_id`.                   |
| `503`       | `service_unavailable`  | A Ticto está temporariamente indisponível. Implemente retry com backoff exponencial.            |

## Erros de validação

Quando a API retorna status `422`, o array `details` traz um objeto por campo com problema, facilitando a exibição de mensagens específicas no seu formulário ou no log da integração.

```json Exemplo de erro 422 com múltiplos campos
{
  "error": "validation_error",
  "message": "Os dados enviados contêm inconsistências. Verifique o campo details.",
  "request_id": "req_2c8b47fa91de",
  "details": [
    {
      "field": "name",
      "message": "O campo name é obrigatório."
    },
    {
      "field": "price",
      "message": "O preço deve ser um número inteiro em centavos e no mínimo 500."
    },
    {
      "field": "charge_type",
      "message": "O valor informado não é válido. Use: one_time ou recurring."
    }
  ]
}
```

## Lidando com rate limit

Quando o limite de requisições é atingido, a API retorna status `429` com o header `Retry-After` indicando quantos segundos aguardar até o reset da cota. Não faça novas tentativas antes desse prazo — elas serão rejeitadas e não irão consumir a fila.

```json Exemplo de erro 429
{
  "error": "too_many_requests",
  "message": "Você atingiu o limite de requisições. Aguarde antes de tentar novamente.",
  "request_id": "req_9a1d05c3e72f",
  "details": []
}
```

O header `Retry-After` estará presente na resposta:

```
Retry-After: 47
```

Consulte a página de [Rate Limits](/api-reference/rate-limits) para detalhes sobre os limites e boas práticas de retry.

## Serviço indisponível

Um status `503` indica que a Ticto está temporariamente fora do ar para manutenção ou por instabilidade pontual. Nesses casos:

- **Não tente imediatamente.** Use backoff exponencial com jitter para espaçar as tentativas.
- Aguarde pelo menos 5 segundos antes da primeira nova tentativa.
- Após 3 a 5 tentativas sem sucesso, registre o erro e notifique sua equipe incluindo o `request_id`.

<Tip>
  Sempre use o campo `error` (slug) para lógica de tratamento no seu código — por exemplo, `if (error === "unauthenticated")`. Use `message` apenas para exibir ao usuário ou para debugging, pois o texto pode mudar entre versões da API.
</Tip>
